# 이동 가능한 프로젝트와 라벨 검토 흐름 개선 설계

## 배경

BBox는 현재 프로젝트의 라벨, 박스, 이미지 상태를 `project.bbox.json`에 저장하지만 원본 이미지는 PC의 절대 경로로 참조한다. 프로젝트 파일은 AppData 내부 프로젝트 라이브러리에 묶여 있어 파일 하나를 다른 PC로 복사한 뒤 제품 UI에서 가져오는 흐름이 없다. 또한 원본 경로가 없는 경우 현재 검증 로직은 이미지 상태를 `error`로 바꿀 수 있어, 다른 PC에서 기존 `confirmed` 또는 `needsReview` 상태를 보존하기 어렵다.

박스 번호는 캔버스와 우측 목록에서 좌상단에서 우하단으로 행별 정렬되지만, 라벨 지정 후 다음 박스를 선택하는 컨트롤러 로직은 박스 생성 순서를 사용한다. 따라서 화면에 보이는 번호와 실제 라벨링 진행 순서가 어긋날 수 있다.

확정 박스는 흰색 테두리 주변의 진한 검정 그림자로 인해 객체가 어둡게 보일 수 있다. 검토가 필요한 자동 라벨은 우측 상세의 제한된 높이 안에 비대화형 점수 행으로 표시되어, 후보 선택과 Enter 확정 방식을 사용자가 발견하기 어렵다.

## 목표

- 이미지가 포함되지 않은 프로젝트 파일을 다른 PC로 옮겨 같은 라벨링 데이터를 확인하고 이어서 작업할 수 있게 한다.
- 저장된 절대 이미지 경로가 존재하면 별도 질문 없이 그대로 사용한다.
- 이미지가 없을 때 파일 또는 폴더 단위로 원본을 다시 연결할 수 있게 한다.
- 원본 파일 가용성과 라벨링 상태를 분리해 이동 과정에서 확정 상태와 annotation을 보존한다.
- 박스 표시 번호와 라벨 지정 후 다음 선택 순서를 동일한 좌상단→우하단 행별 정렬로 통합한다.
- 확정 박스의 흰색 테두리를 유지하면서 검정 그림자와 내부 가림을 제거한다.
- 우측 상세에서 추천 라벨을 명확히 선택하고 Enter 또는 버튼으로 적용할 수 있게 한다.

## 범위 밖

- 프로젝트 파일에 원본 이미지 포함
- 클라우드 동기화 또는 다중 사용자 동시 편집
- 세션 Undo/Redo 기록 전송
- 이미지 뷰어의 확대, 패닝, 임시 선택 상태 전송
- 프로젝트를 열 때 원본 이미지 파일 자체를 수정하거나 이동

## 제품 흐름

### 프로젝트 파일 저장

작업 화면에 `프로젝트 파일 저장` 동작을 제공한다. 이 동작은 자동 저장 대상인 내부 프로젝트를 다른 위치로 옮기는 `Save As`가 아니라, 현재 메모리 상태를 전송 가능한 `.bbox.json` 사본으로 기록한다.

저장 전 예약된 자동 저장 작업을 완료하고 현재 프로젝트의 라벨, 이미지 manifest, 박스, 확정 상태, 자동 라벨 메타데이터를 직렬화한다. 전송 파일에는 원본 이미지 바이트를 포함하지 않는다. 전송 파일의 `projectFilePath`는 다른 PC의 내부 경로로 오인되지 않도록 null로 기록한다. 현재 PC의 프로젝트는 기존 내부 경로에서 계속 자동 저장된다.

대상 파일이 이미 있으면 덮어쓰기 확인을 받는다. 쓰기는 가능한 범위에서 임시 파일 작성 후 교체 방식으로 처리해 중간 실패로 불완전한 JSON이 남지 않게 한다.

### 프로젝트 파일 가져오기

시작 화면에 `프로젝트 파일 가져오기` 동작을 제공한다. 사용자가 `.bbox.json` 파일을 선택하면 다음 순서로 처리한다.

1. JSON 형식과 schema version을 검증하고 현재 version으로 migration한다.
2. 프로젝트 데이터 자체의 필수 필드와 참조 무결성을 검증한다.
3. 새 PC의 내부 프로젝트 라이브러리에 새로운 프로젝트 ID와 내부 경로를 만든다.
4. 전송 파일의 `projectFilePath` 대신 새 내부 경로를 설정해 저장한다.
5. 프로젝트 이름이 기존 항목과 같아도 데이터를 덮어쓰지 않고 별도 프로젝트로 가져온다.
6. 저장된 각 이미지 절대 경로의 현재 PC 가용성을 확인한다.
7. 모든 원본이 존재하면 곧바로 작업 화면을 연다. 누락된 원본이 있으면 라벨링 상태를 유지한 채 재연결 안내를 표시한다.

수정된 프로젝트를 다시 전달하려면 사용자가 현재 프로젝트에서 새 프로젝트 파일 사본을 저장한다.

## 원본 이미지 재연결

### 원칙

원본 이미지 가용성은 `ImageStatus`와 분리한다. 다른 PC에서 파일을 찾을 수 없다는 이유로 기존 `confirmed`, `needsReview`, `queued` 상태를 `error`로 덮어쓰지 않는다. `AppController`가 이미지 ID별 `SourceAvailability` 값을 현재 환경의 파생 상태로 관리하며 이 값은 프로젝트 JSON에 저장하지 않는다. 박스, 라벨, 자동 라벨 후보, 확정 여부는 재연결과 무관하게 보존한다.

작업 화면은 누락 이미지 수, 원래 경로, 원래 가져온 위치를 보여주고 다음 두 동작을 제공한다.

- `파일로 다시 연결`: 하나 또는 여러 이미지 파일을 선택한다.
- `폴더로 다시 연결`: 선택한 폴더와 하위 폴더를 검색한다.

프로젝트가 파일 추가와 폴더 추가를 혼합해 구성되었어도 두 동작을 함께 사용할 수 있다. 성공한 항목은 즉시 새 경로로 갱신하고, 찾지 못한 항목은 누락 상태로 남긴다. 부분 성공은 전체 작업을 rollback하지 않는다.

### 매칭 우선순위

폴더 재연결은 `importedFrom`별 그룹과 기존 상대 경로를 먼저 사용한다. 상대 경로가 일치하지 않으면 아래 검증 정보를 사용한다.

1. 저장된 `contentSha256`가 있으면 hash가 정확히 일치하는 후보만 허용
2. hash가 없으면 기존 파일명, width, height가 모두 일치하는 후보만 허용

파일 재연결도 같은 검증 순서를 사용한다. 한 누락 이미지에 여러 후보가 남거나 하나의 새 파일이 여러 이미지에 배정될 수 있으면 자동 연결하지 않는다. 해당 항목을 미해결 상태로 두고 사용자가 명시적으로 선택하게 한다. 해시가 없는 기존 프로젝트에서도 파일명과 해상도가 유일하게 일치할 때만 자동 연결한다.

재연결이 성공하면 `sourcePath`를 새 절대 경로로 바꾸고 `importedFrom`을 선택 방식에 맞게 갱신한다. 변경 결과는 새 PC의 내부 프로젝트에 자동 저장한다.

## 박스 표시 순서와 순차 라벨링

좌상단→우하단 행별 순서를 계산하는 공통 `BoxDisplayOrder` 책임을 annotation 계층에 둔다. UI 전용 helper에 정렬을 중복하지 않는다.

정렬은 다음 규칙을 사용한다.

1. 삭제되지 않은 박스를 y 좌표, x 좌표, 기존 순서로 먼저 안정 정렬한다.
2. 정렬된 첫 박스의 y와 height를 행 anchor로 고정하고, 다음 박스의 y 차이가 `max(4px, min(anchorHeight, boxHeight) × 0.5)` 이하일 때만 같은 행으로 그룹화한다. 새 행이 시작되면 그 첫 박스를 새 anchor로 사용한다.
3. 같은 행의 박스는 x 좌표 오름차순으로 정렬한다.
4. 좌표가 같은 경우 기존 박스 순서와 ID 순으로 안정적인 tie-break를 적용한다.

동일한 정렬 결과를 다음 위치에서 사용한다.

- 캔버스 박스 번호
- 우측 박스 목록 번호와 순서
- 표 보기 순서
- 라벨 지정 후 다음 미라벨 박스 선택

선택된 박스에 라벨을 적용하면 표시 번호가 더 큰 미라벨 박스 중 가장 앞 박스를 선택한다. 이후 미라벨 박스가 없고 앞 번호에 미라벨 박스가 남아 있으면 가장 앞 번호로 순환한다. 모든 박스가 라벨링되면 현재 선택을 유지한다. 이 규칙은 수동 박스와 자동 박스에 동일하게 적용한다.

## 확정 박스 표시

확정된 라벨 박스의 기본 테두리는 현재와 같이 흰색 2px을 유지한다. 객체 가시성을 위해 기존 진한 검정 `BoxShadow`를 제거하고 박스 내부 채움은 투명하게 유지한다.

라벨 대표 색은 라벨명 배지와 우측 박스 목록의 상태 점에 사용한다. 선택된 박스는 흰색 테두리를 더 굵게 표시하고 선택 강조선과 resize handle로 구분한다. 선택 여부를 내부의 어두운 색 채움으로 표현하지 않는다.

검토 필요 박스의 경고 테두리와 상태 표시는 유지하되, 내부 채움은 객체 확인을 방해하지 않는 낮은 불투명도를 사용한다.

## 추천 라벨 검토 UX

선택된 박스 상세에서 검토 후보를 고정된 126px 영역 안에 가두지 않는다. 우측 inspector의 기존 전체 스크롤 흐름 안에서 상세 영역이 필요한 높이를 사용하도록 한다.

검토가 필요한 박스에는 다음 요소를 표시한다.

- 제목: `추천 라벨을 선택한 뒤 Enter를 누르세요`
- 각 후보의 라벨 색상, 라벨명, 신뢰도
- 현재 선택된 후보 표시
- `선택한 라벨 적용 · Enter` 버튼
- 추천 후보가 없을 때 `추천 결과 없음`과 빠른 라벨 버튼/단축키 안내

첫 번째 후보를 기본 선택한다. 사용자는 후보 행을 클릭하거나 `↑/↓` 키로 선택을 바꾼다. 텍스트 입력에 focus가 없고 검토 필요 박스가 선택된 상태에서 Enter를 누르면 현재 선택된 후보를 사용자 확정 라벨로 적용한다. 버튼 클릭도 같은 controller action을 호출한다.

박스를 바꾸면 후보 선택은 새 박스의 첫 후보로 초기화한다. 후보 적용 후에는 `BoxDisplayOrder`에 따라 다음 미라벨 박스를 선택한다. 숫자와 문자 빠른 라벨 단축키는 기존처럼 선택 박스에 직접 라벨을 적용하고 같은 다음 선택 규칙을 사용한다.

우측 박스 목록의 상태 문구는 행동을 설명하도록 바꾼다.

- 일반 proposal: `라벨 선택 필요`
- 자동 추천 검토 대상: `추천 검토 필요`
- 확정 라벨: 라벨명
- 유효하지 않은 박스: 오류 문구

## 오류 처리

### 프로젝트 저장 및 가져오기

- JSON 파싱 실패: 프로젝트를 만들지 않고 파일 형식 오류를 안내한다.
- 지원하지 않는 schema version: 현재 앱 버전으로 열 수 없음을 안내한다.
- 내부 프로젝트 저장 실패: 부분적으로 생성된 index 항목을 남기지 않고 재시도를 제공한다.
- 프로젝트 파일 저장 실패: 현재 내부 프로젝트와 메모리 변경을 유지하고 대상 경로와 권한 확인을 안내한다.
- 기존 대상 파일: 명시적인 덮어쓰기 확인 없이 교체하지 않는다.

### 원본 재연결

- 선택한 파일이나 폴더 접근 실패: 기존 경로와 annotation을 유지한다.
- 일부만 매칭: 연결 성공 수, 미해결 수, 중복 후보 수를 요약한다.
- 모호한 후보: 자동 연결하지 않고 사용자가 직접 연결할 수 있게 한다.
- 손상되거나 decode할 수 없는 이미지: sourcePath를 확정하기 전에 유효성을 검증하고 실패 항목을 미해결로 유지한다.
- 재연결 저장 실패: 메모리에서 매칭 결과를 잃지 않고 저장 재시도를 제공한다.

## 구조

### project

`ProjectSnapshotService`를 추가해 내부 자동 저장과 전송 파일 저장을 분리한다. snapshot 저장은 현재 프로젝트의 `projectFilePath`를 변경하지 않는다. 가져오기는 `ProjectStore`의 migration과 validation을 재사용한 뒤 `ProjectLibrary`에 새 프로젝트를 생성한다.

`SourceRelinkService`는 파일 시스템 탐색, 후보 검증, 매칭 결과를 담당한다. UI는 파일 또는 폴더 선택과 결과 표시만 담당하며 모델을 직접 수정하지 않는다.

### annotation

`BoxDisplayOrder`는 삭제되지 않은 박스의 안정적인 표시 순서를 반환한다. 컨트롤러와 workbench UI가 이 API를 함께 사용한다.

### ui

`AppController`는 프로젝트 파일 저장/가져오기, source availability, 재연결 결과, 추천 후보 선택과 적용 action을 노출한다. 프로젝트 저장과 긴 파일 탐색은 비동기로 처리하고 진행 중에는 중복 실행을 막는다.

시작 화면에는 `프로젝트 파일 가져오기`, 작업 화면 프로젝트 메뉴에는 `프로젝트 파일 저장`을 제공한다. 원본 누락 안내는 사용자가 작업 데이터를 잃었다고 오해하지 않도록 `라벨링 데이터는 보존되어 있습니다` 문구와 파일/폴더 재연결 action을 함께 보여준다.

## 테스트

### 단위 테스트

- snapshot 저장 시 현재 프로젝트의 내부 `projectFilePath`가 바뀌지 않음
- snapshot save/load에서 라벨, 박스, 확정 상태, 자동 라벨 메타데이터 보존
- 가져온 프로젝트가 새 내부 경로를 사용하고 원본 절대 경로를 우선 확인
- 파일 재연결과 폴더 재연결의 성공, 부분 성공, 미해결 결과
- hash 우선 매칭과 파일명/해상도 fallback
- 중복 후보를 자동 연결하지 않음
- 원본 누락이 `ImageStatus`를 바꾸지 않음
- `BoxDisplayOrder`의 행별 정렬, tolerance, tie-break 안정성
- 라벨 지정 후 화면 번호 순서로 다음 미라벨 박스 선택 및 순환

### 위젯 테스트

- 시작 화면에서 프로젝트 파일 가져오기 가능
- 작업 화면에서 프로젝트 파일 저장 가능
- 누락 원본 안내에 파일과 폴더 재연결 action이 모두 표시됨
- 부분 재연결 결과와 남은 누락 수 표시
- 확정 박스가 흰색 테두리를 유지하고 검정 그림자와 내부 채움이 없음
- 추천 후보가 우측 상세에서 잘리지 않음
- 후보 클릭, 방향키, Enter, 적용 버튼이 동일한 라벨 적용 action 수행
- 후보 적용 후 다음 표시 번호의 미라벨 박스가 선택됨
- 1280x720 화면에서 후보 적용 버튼과 재연결 action 접근 가능

### 통합 테스트

- 프로젝트 생성→이미지 파일 및 폴더 추가→라벨링→확정→snapshot 저장→새 라이브러리 가져오기→상태 일치
- 동일 절대 경로가 있는 환경에서 추가 질문 없이 작업 화면 진입
- 원본 경로가 없는 환경에서 annotation 상태를 보존한 채 파일/폴더 재연결
- 재연결 후 프로젝트 재열기와 COCO export에서 image ID, category ID, bbox 유지
- 수동 및 자동 박스가 섞인 이미지에서 표시 번호와 순차 라벨링 순서 일치

## 완료 기준

- 이미지 없는 `.bbox.json` 파일 하나로 프로젝트 데이터를 다른 PC에 전달할 수 있다.
- 원본 절대 경로가 존재하면 즉시 작업을 이어갈 수 있다.
- 원본이 없으면 파일과 폴더 두 방식으로 재연결할 수 있다.
- 원본 누락과 재연결 과정에서 라벨, 박스, 확정 상태가 바뀌지 않는다.
- 화면 번호와 라벨 지정 후 다음 선택이 좌상단→우하단 행별 순서로 일치한다.
- 확정 박스는 흰색 테두리이며 진한 검정 그림자와 내부 가림이 없다.
- 우측 상세에서 추천 후보를 명확히 선택하고 Enter 또는 버튼으로 적용할 수 있다.
- 관련 단위, 위젯, 통합 테스트가 통과한다.

## 자체 검토

이 설계는 이미지 미포함, 기존 절대 경로 우선 사용, 파일 및 폴더 재연결, 흰색 확정 박스 테두리라는 승인된 요구사항을 반영한다. 원본 파일 가용성을 라벨링 상태와 분리해 PC 이동이 annotation을 훼손하지 않게 했고, 표시 번호와 순차 라벨링이 동일한 정렬 구현을 사용하도록 경계를 통합했다. 추천 라벨 UX는 마우스와 키보드를 모두 지원하며 작은 데스크톱 화면의 접근성 검증을 포함한다.
