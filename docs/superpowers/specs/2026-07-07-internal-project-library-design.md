# Internal Project Library Design

## Context

The current app saves and opens `.bbox.json` project files through file picker
dialogs. That works technically, but it makes project selection feel like file
management instead of an in-app workflow.

The new requirement is to save projects automatically inside the software's own
storage area and let users choose projects from inside the app. The selected
direction is Windows AppData storage. Project data is internal; source images
are not copied and continue to be referenced from the user-selected image
folder.

## Product Scope

This feature replaces the primary start experience with an internal project
home:

- create a project without choosing a save file path
- list saved projects in the app
- open a project from the in-app list
- automatically save changes to the internal project file
- rename and delete projects from the project home
- detect when the referenced image folder is missing
- reconnect a missing image folder without losing annotations

Out of scope for this feature:

- copying original images into AppData
- cloud sync, accounts, or collaboration
- project templates
- import/export of complete project backup bundles
- migration from arbitrary external `.bbox.json` files unless added later as a
  separate import tool

COCO export remains a separate user-selected output file flow.

## Storage Layout

On Windows, the app stores project data under:

```text
%APPDATA%\BBoxLabeler\
```

The project library layout is:

```text
%APPDATA%\BBoxLabeler\projects\index.json
%APPDATA%\BBoxLabeler\projects\<project-id>\project.bbox.json
```

`<project-id>` is generated by the app and never edited by the user. It should
be stable, filesystem-safe, and collision-resistant enough for local use. A
timestamp plus a short sanitized project-name slug is sufficient for MVP, for
example:

```text
20260707T143012-bbox-project
```

The app does not modify or copy source images. `project.bbox.json` keeps the
existing `imageFolderPath`, while each image keeps its folder-relative path.
If the image folder moves, the project remains openable but enters a reconnect
state before image viewing can work normally.

## Project Index

`index.json` is a lightweight catalog for the start screen. It is not the source
of annotation truth. Annotation truth remains in each `project.bbox.json`.

Recommended index shape:

```json
{
  "schemaVersion": 1,
  "projects": [
    {
      "id": "20260707T143012-bbox-project",
      "name": "BBox Project",
      "projectFilePath": "C:\\Users\\...\\AppData\\Roaming\\BBoxLabeler\\projects\\20260707T143012-bbox-project\\project.bbox.json",
      "imageFolderPath": "D:\\datasets\\images",
      "createdAt": "2026-07-07T05:30:12.000Z",
      "updatedAt": "2026-07-07T05:45:10.000Z",
      "imageCount": 120,
      "confirmedImageCount": 40,
      "errorImageCount": 1
    }
  ]
}
```

The index is updated after project creation, project save, rename, delete, image
import, and image confirmation. If the index is missing or corrupted, the app
can rebuild a best-effort index by scanning `projects/*/project.bbox.json`.

## Architecture

Keep the existing `ProjectStore` as the low-level UTF-8 JSON serializer for a
single project file. Add a higher-level project library service in
`lib/project`.

Recommended modules:

- `ProjectStore`: unchanged responsibility; save/load one `AnnotationProject`
  at a provided path.
- `ProjectLibrary`: owns the AppData root, project IDs, index read/write,
  project creation paths, rename/delete, and list metadata.
- `ProjectLibraryEntry`: a small immutable metadata model used by the start
  screen.
- `AppController`: gains library-aware methods such as
  `createLibraryProject(name)`, `openLibraryProject(id)`,
  `renameCurrentProject(name)`, and `deleteLibraryProject(id)`.
- `StartScreen` or a new `ProjectHomeScreen`: displays the project list and
  project actions instead of opening arbitrary project files.

The controller should still expose only explicit actions. Widgets should not
mutate project JSON or index files directly.

## User Flow

### First Launch

The start screen shows an empty project home with a project name field and a
primary `New project` action. Creating a project immediately creates the
internal directory and saves an empty `project.bbox.json` there. The user then
enters the workbench and selects an image folder.

### Returning Launch

The start screen loads `index.json` and shows projects sorted by `updatedAt`
descending. Each project row or card shows:

- project name
- last updated time
- image count
- confirmed count
- error count
- image folder status

Selecting a project opens it directly from its internal path. No file dialog is
needed.

### Missing Image Folder

When opening a project, the app checks whether `imageFolderPath` exists. If it
does not, the project still opens, but the UI shows a clear missing-folder
state:

```text
Image folder not found.
```

The primary action is `Reconnect image folder`. The user picks or enters a new
folder. The app verifies that known relative image paths can be found. If some
files are still missing, it shows a concise warning and allows the user to
continue, because annotations may still be valuable.

### Rename

Renaming a project updates `AnnotationProject.name`, saves the project file,
and updates the matching index entry. It does not rename the project directory;
the stable project ID remains the directory name.

### Delete

Deleting a project removes only the internal project directory and its index
entry. It never deletes the original image folder. The UI must confirm this
action and explicitly say that source images will not be removed.

## Auto Save

Current autosave behavior should become the normal save path for all projects.
Because every library project has a `projectFilePath` from creation, changes can
be saved automatically immediately after:

- image folder import
- box create, move, resize, delete
- label create, rename, or delete
- label assignment
- image confirmation
- project rename

Manual `Save project` can remain as a toolbar action, but it should save to the
internal project file without opening a save dialog. The label can be `Save` or
it can be hidden if autosave status is shown clearly.

After each successful save, `ProjectLibrary` refreshes the index metadata for
the current project so the project home stays accurate.

## Error Handling

Handle these failures explicitly:

- AppData root cannot be created
- `index.json` cannot be read
- `index.json` has an unsupported schema version
- a listed project file is missing
- a project file has an unsupported project schema version
- a project directory cannot be deleted
- autosave fails
- the referenced image folder is missing
- reconnected image folder does not contain all expected relative image files

If `index.json` is unreadable, the app should try to rebuild it from project
folders. If rebuild fails, show a concise error and keep the user on the project
home with the ability to create a new project.

If autosave fails, do not discard in-memory changes. Keep the project open,
surface the error, and allow retry.

## Testing

Unit tests:

- resolves the AppData project root using an injectable base directory
- creates a project directory and `project.bbox.json`
- writes and reads `index.json`
- lists projects sorted by `updatedAt`
- rebuilds index from project directories
- updates index metadata after saving
- renames project without changing project ID
- deletes only internal project files, not image folders
- detects missing image folder
- reconnects image folder while preserving relative image paths

Widget tests:

- first launch shows empty project home
- creating a project opens the workbench without save-file dialog
- returning launch lists saved projects
- tapping a project opens it
- missing image folder state shows reconnect action
- deleting a project asks for confirmation and removes it from the list

Integration test:

- create internal project
- import image folder
- label and confirm one image
- close by creating a fresh controller/app instance
- project appears in project home
- open it from the in-app list
- verify labels, image statuses, boxes, and image folder path are restored

## Migration And Compatibility

This feature does not need to change the existing project JSON schema. The
project file can keep `schemaVersion: 1` because the internal library behavior
is represented by storage location and `index.json`.

Existing direct file open/save dialogs should be removed from the primary UX.
If external project import is needed later, it should be designed as an explicit
`Import project` action that copies the external `.bbox.json` into the
internal project library and creates an index entry.

## UX Notes

The project home is a work screen, not a marketing landing page. It should be
compact and action-oriented:

- left or top area: create project
- main area: project list
- row actions: open, rename, delete
- status text: last updated, image count, confirmed count, missing folder

Avoid long explanatory text. Empty and error states should tell the user the
next action directly.

## Self Review

No incomplete markers remain. The selected storage strategy is AppData project data
with source image references. The design keeps `ProjectStore` focused on one
project file and adds `ProjectLibrary` as the catalog layer. It avoids image
copying, cloud features, and external project import in this scope. Missing
image folders, autosave failures, deletion safety, testing, and compatibility
are explicit.
