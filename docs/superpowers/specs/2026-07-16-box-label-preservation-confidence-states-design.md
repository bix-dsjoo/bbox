# 박스 라벨 보존 및 신뢰도 상태 설계

## 목표

박스를 이동하거나 크기를 조절할 때 사용자가 지정한 라벨이 사라지는 문제를 수정한다. 자동 분류 결과는 최상위 라벨 후보의 분류 신뢰도를 단일 기준으로 사용해 다음 세 상태로 표시한다.

- `confidence >= 0.98`: 라벨 자동 지정, 흰색 테두리
- `0.50 < confidence < 0.98`: 추천 라벨 검토 필요, 빨간색 테두리
- `confidence <= 0.50`: 추천 라벨을 지정하지 않은 미라벨, 회색 테두리

경계값은 0.50을 미라벨 구간에, 0.98을 자동 지정 구간에 포함한다. 여기서 confidence는 검출기 박스 신뢰도가 아니라 분류기의 최상위 `LabelCandidate.score`다.

## 조사 결과와 원인

`AppController._editSelectedBox`는 박스에 자동화 메타데이터가 하나라도 남아 있으면 편집 직후 `status`, `labelId`, `labelSource`, `automation`을 모두 초기화한다. 수동 라벨 지정은 `labelSource`를 `user`로 변경하지만 후보 이력 보존을 위해 `automation` 객체를 남기므로, 자동 박스에 수동 라벨을 지정한 뒤 좌표를 수정하면 수동 라벨까지 자동 라벨로 오인되어 삭제된다.

자동 분류 상태는 현재 Python 정책이 manifest의 confidence, margin, 보수 클래스, 박스 품질 사유를 함께 사용해 `accepted` 또는 `review`로 결정한다. 따라서 요청한 세 구간을 정확히 보장하려면 UI 색만 바꾸는 것이 아니라 분류 정책의 상태 결정부터 일관되게 변경해야 한다.

## 검토한 접근 방식

### 1. 분류 정책을 단일 진실 공급원으로 변경 — 채택

Python 분류 정책이 최상위 후보 점수로 `accepted`, `review`, `unavailable`을 결정하고 Flutter는 기존 프로토콜을 그대로 도메인 상태로 변환한다. 상태 저장, 캔버스 색상, 오른쪽 상세, export 판정이 모두 동일한 결과를 사용한다.

장점은 정책 중복이 없고 저장 후 다시 열어도 상태가 일관된다는 것이다. Python 정책 테스트와 Dart 프로토콜·UI 테스트가 모두 필요하다.

### 2. Flutter에서 worker 응답을 신뢰도 기준으로 덮어쓰기

worker 응답은 유지하고 Dart 파서가 후보 점수를 다시 해석한다. 빠르게 적용할 수 있지만 Python과 Dart가 서로 다른 정책을 갖게 되고 로그와 실제 화면 상태가 어긋난다. 채택하지 않는다.

### 3. 캔버스 색상만 신뢰도로 변경

데이터 상태는 유지하면서 선 색만 바꾼다. 빨간 박스가 내부적으로 확정 처리되는 등 저장·확정·export와 화면이 불일치하므로 채택하지 않는다.

## 상세 동작

### 박스 편집과 라벨 보존

- `BoxStatus.labeled`이고 실제 `labelId`가 있는 박스는 라벨 출처가 `user` 또는 `auto`인지와 관계없이 이동·크기 조절·좌표 직접 입력 후에도 `status`, `labelId`, `labelSource`, 후보 이력을 항상 보존한다.
- 이미 라벨이 지정된 흰색 박스는 편집 후 자동 재분류하지 않는다. 따라서 백그라운드 결과가 기존 라벨을 바꾸거나 박스를 미라벨로 되돌리는 일도 없다.
- 미라벨 및 검토 필요 박스만 편집 후 재분류하며, 결과가 도착하기 전까지 현재 회색 또는 빨간색 상태를 유지한다.
- 미라벨 박스의 자동 재분류 결과는 최신 박스 좌표에 대한 결과일 때만 세 구간 정책을 적용한다.
- 사용자가 재분류 대기 중 라벨을 직접 지정하면 기존 규칙대로 사용자 라벨이 자동 결과보다 우선하고, 이후 도착한 자동 결과는 적용하지 않는다.
- 모든 실제 좌표 편집은 이미지 상태를 `needsReview`로 변경한다. 박스 라벨 보존과 이미지 전체 확정 상태는 별개다.

### 신뢰도 상태

- 0.98 이상은 `BoxStatus.labeled`, 실제 `labelId`, `LabelSource.auto`로 저장한다.
- 0.50 초과 0.98 미만은 `BoxStatus.proposal`, 실제 `labelId == null`, `suggestedLabelId`와 후보 목록을 저장한다. 이 상태는 `requiresLabelReview == true`이며 빨간색이다.
- 0.50 이하는 `BoxStatus.proposal`, `labelId == null`, `suggestedLabelId == null`로 저장한다. 후보 목록은 오른쪽 상세 표시를 위해 보존할 수 있지만 자동 선택 후보로 취급하지 않으며 회색이다.
- 0.98 이상이라는 조건은 자동 지정의 최종 상태 기준이다. 기존 margin, conservative class, edge/duplicate 검토 사유는 진단 메타데이터로 남길 수 있지만 0.98 이상 결과를 빨간색으로 강등하지 않는다.
- 분류기를 사용할 수 없거나 유효한 후보 점수가 없으면 미라벨·회색 상태로 처리한다.

## 데이터와 호환성

기존 `accepted`, `review`, `unavailable` worker 프로토콜과 `BoundingBox` 저장 구조를 재사용한다. 프로젝트 파일 형식 버전은 변경하지 않는다. 기존 프로젝트에 저장된 박스 상태는 재분류 전까지 그대로 유지하며, 편집 또는 명시적 자동 분류 때 새 정책이 적용된다.

manifest에는 정책값을 명시적으로 `0.98`과 `0.50`으로 기록해 실행 환경과 테스트가 같은 기준을 사용하게 한다. 현재 manifest 파일에는 사용자의 미커밋 변경이 있으므로 해당 변경을 보존하면서 필요한 키만 최소 수정한다.

## 테스트

- 자동화 이력이 있는 사용자 라벨 박스를 편집해도 라벨과 출처가 유지되는 회귀 테스트
- 수동 및 자동 라벨 박스가 편집 후 흰색·라벨·출처를 유지하고 재분류를 요청하지 않는 테스트
- 정확히 0.50, 0.50 초과, 0.98 미만, 정확히 0.98의 정책 경계 테스트
- 고신뢰 결과가 자동 라벨, 중간 결과가 검토 제안, 저신뢰 결과가 제안 없는 미라벨로 변환되는 worker 및 Dart 파서 테스트
- 회색·빨간색·흰색 캔버스 표시와 이미지 확정 가능 조건 테스트
- 관련 Python 테스트, Flutter 단위·위젯 테스트, 정적 분석을 실행한다.

## 제외 범위

모델 재학습, 라벨 종류 변경, 검출기 신뢰도 조정, 박스 정렬 규칙 변경은 포함하지 않는다.
