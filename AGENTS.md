# AGENTS.md

이 문서는 Flutter 기반 객체 검출 학습 데이터 라벨링 프로그램을 개발하는 에이전트와 기여자를 위한 기준 문서다. 제품 목표, 핵심 UX, 데이터 모델, 구현 원칙, 검증 기준을 함께 정의한다.

## 프로젝트 목표

사용자가 촬영한 이미지 폴더를 불러온 뒤, 객체 검출 모델 학습용 Bounding Box 라벨 데이터를 COCO 형식으로 빠르고 정확하게 만들 수 있는 프로그램을 개발한다.

핵심 가치는 다음과 같다.

- 반복 라벨링 작업을 최대한 빠르게 만든다.
- 처음 쓰는 사용자도 프로젝트 생성, 이미지 불러오기, 박스 확인, 라벨 지정, 수정, 저장, export 흐름을 자연스럽게 이해하게 만든다.
- AI 또는 알고리즘 기반 후보 박스는 작업을 돕는 제안일 뿐이며, 최종 데이터 품질은 사용자가 쉽게 검토하고 확정할 수 있어야 한다.
- COCO export는 미확정 이미지가 남아 있어도 가능해야 하며, 내보내기 전 누락/오류를 명확히 알려준다.

## 권장 제품 범위

### MVP

- 새 프로젝트 생성
- 이미지 폴더 선택 및 이미지 목록 생성
- 이미지별 후보 Bounding Box 자동 생성
- 이미지 목록에 처리/미확정/확정/오류 상태 표시
- 이미지 뷰어에서 후보 박스 오버레이 표시
- 박스 목록과 이미지 오버레이 간 양방향 선택
- 박스 생성, 선택, 이동, 크기 조절, 삭제
- 라벨 생성 및 박스에 라벨 매칭
- 라벨 매칭 시 대표 색상으로 박스 색 변경
- 모든 박스가 라벨링되면 이미지 확정 가능
- 확정/미확정 상태와 무관하게 COCO export 가능
- 자동 저장 및 프로젝트 재열기

### MVP 이후

- 탐지 모델 선택 또는 교체
- 클래스별 자동 라벨 추천
- 이전 이미지 라벨/박스 복사
- 단축키 커스터마이징
- COCO train/valid/test split export
- 라벨링 통계와 품질 검토 화면
- 다중 사용자 협업, 클라우드 동기화, 계정 기능

MVP에서는 클라우드, 계정, 협업 서버, 모델 학습 기능을 만들지 않는다. 이 앱은 우선 로컬 데이터셋 제작 도구다.

## 대상 플랫폼

Flutter 앱으로 개발한다. 파일 시스템 접근과 폴더 선택이 중요한 제품이므로 MVP는 데스크톱 사용성을 우선한다.

- 우선순위: Windows desktop
- 고려 대상: macOS, Linux
- 모바일/태블릿은 후속 확장으로 본다.

플랫폼별 파일 권한, 긴 경로, 한글 파일명, 대용량 이미지 폴더 처리를 반드시 고려한다.

## 핵심 사용자 흐름

1. 사용자가 앱을 실행한다.
2. 새 프로젝트를 만든다.
3. 이미지가 들어 있는 폴더를 선택한다.
4. 앱이 지원 이미지 파일을 스캔하고 이미지 목록을 만든다.
5. 앱이 이미지별로 순차 또는 백그라운드 큐 방식으로 후보 박스를 생성한다.
6. 후보 박스가 생성된 이미지는 목록에 `미확정` 상태로 표시된다.
7. 사용자가 이미지를 선택하면 이미지 위에 박스가 오버레이되고, 오른쪽 또는 보조 패널에 박스 목록이 표시된다.
8. 사용자는 박스를 클릭하거나 목록에서 선택한다.
9. 박스가 객체와 맞지 않으면 위치/크기를 조정하거나 삭제한다.
10. 누락된 객체가 있으면 사용자가 직접 새 박스를 그린다.
11. 박스가 실제 객체와 맞으면 라벨을 지정한다.
12. 원하는 라벨이 없으면 즉시 새 라벨을 만든다.
13. 라벨이 지정된 박스는 해당 라벨 대표 색으로 표시된다.
14. 회색 후보 박스가 없고 유효한 라벨 박스만 남으면 `확정` 버튼이 활성화된다.
15. 사용자가 확정하면 이미지 목록 상태가 `확정`으로 바뀐다.
16. 모든 이미지를 반복해서 검토한다.
17. 사용자는 언제든 COCO 형식으로 export할 수 있다.

## 상태 모델

### 프로젝트 상태

- `empty`: 프로젝트 생성 전
- `ready`: 이미지 폴더가 연결되고 작업 가능
- `scanning`: 이미지 파일 스캔 중
- `detecting`: 후보 박스 생성 중
- `dirty`: 저장되지 않은 변경 사항 있음
- `exporting`: COCO 파일 생성 중
- `error`: 복구 가능한 오류 발생

### 이미지 상태

- `queued`: 아직 후보 생성 전
- `detecting`: 후보 생성 중
- `needsReview`: 후보 박스 검토 필요
- `confirmed`: 사용자가 검토 완료
- `error`: 이미지 로드, 디코딩, 탐지 중 오류

이미지 상태는 라벨링 진행률과 별도로 표시한다. 예를 들어 라벨 박스가 일부 있어도 사용자가 확정하지 않았으면 `needsReview`다.

### 박스 상태

- `proposal`: AI/알고리즘이 제안한 미라벨 후보 박스. 회색으로 표시한다.
- `selected`: 현재 선택된 박스. 라벨 색상 또는 회색 위에 선택 강조선을 추가한다.
- `labeled`: 라벨이 지정된 유효 박스. 라벨 대표 색으로 표시한다.
- `deleted`: 삭제된 박스. 일반 UI에는 보이지 않지만 undo/redo와 감사 로그를 위해 세션 중 추적할 수 있다.

COCO export에는 `labeled` 박스만 포함한다. `proposal` 박스는 COCO annotation으로 내보내지 않는다.

## UI/UX 원칙

### 첫 사용 경험

- 첫 화면은 프로젝트 생성 또는 기존 프로젝트 열기 행동이 바로 보이게 한다.
- 빈 상태 문구는 짧고 행동 중심이어야 한다.
- 사용자가 다음에 해야 할 일을 화면 구조 자체로 알 수 있어야 한다.
- 긴 설명, 튜토리얼 텍스트, 장식적인 랜딩 페이지를 만들지 않는다.

### 작업 화면 구조

권장 레이아웃은 다음과 같다.

- 왼쪽: 이미지 목록, 상태, 진행률, 검색/필터
- 중앙: 이미지 뷰어와 박스 오버레이
- 오른쪽: 선택 이미지 정보, 박스 목록, 라벨 지정, 박스 속성
- 상단 또는 하단: 프로젝트 저장, 탐지 진행률, export, 확정 버튼

반복 작업에서 마우스 이동 거리가 길어지지 않도록 핵심 액션은 선택 박스 근처 또는 오른쪽 패널에서 빠르게 처리할 수 있게 한다.

### 이미지 목록

이미지 목록에는 최소한 다음 정보를 표시한다.

- 썸네일
- 파일명
- 상태: 대기, 탐지 중, 미확정, 확정, 오류
- 박스 수
- 미라벨 박스 수
- 라벨 완료 박스 수

필터를 제공한다.

- 전체
- 미확정
- 확정
- 오류
- 미라벨 박스 있음

### 이미지 뷰어

이미지 뷰어는 라벨링 작업의 중심이다.

- 확대/축소, 화면 맞춤, 실제 크기 보기
- 패닝
- 박스 클릭 선택
- 드래그로 새 박스 생성
- 선택 박스 이동
- 핸들 드래그로 크기 조정
- Delete 또는 Backspace로 선택 박스 삭제
- Esc로 선택 해제 또는 그리기 취소
- Undo/Redo

좌표 계산은 항상 원본 이미지 픽셀 좌표를 기준으로 한다. 화면 확대/축소나 fit 모드는 표시 좌표에만 영향을 줘야 한다.

### 박스 목록

박스 목록에는 다음 정보를 표시한다.

- 번호 또는 짧은 ID
- 라벨명 또는 `미라벨`
- 상태 색상
- bbox 좌표: x, y, width, height
- 면적

목록에서 항목을 선택하면 이미지 위 박스도 선택된다. 이미지 위 박스를 선택하면 목록도 같은 항목으로 스크롤/선택된다.

### 라벨 관리

- 라벨은 프로젝트 단위로 관리한다.
- 라벨명은 중복되지 않아야 한다.
- 라벨 대표 색상을 저장한다.
- 새 라벨 생성은 라벨 선택 흐름 안에서 바로 가능해야 한다.
- 라벨명 변경 시 이미 매칭된 박스와 export category도 일관되게 반영한다.
- 삭제된 라벨이 기존 박스에 쓰이는 경우 삭제를 막거나 대체 라벨을 선택하게 한다.

### 색상 규칙

- 미라벨 후보 박스: 회색
- 라벨 지정 박스: 라벨 대표 색
- 선택 박스: 기존 색상 위에 명확한 선택 강조선 또는 핸들 표시
- 오류/유효하지 않은 박스: 빨강 계열 경고 표시

색상만으로 상태를 구분하지 말고 텍스트, 아이콘, 패턴, 선택 강조를 함께 사용한다.

### 확정 버튼

이미지 확정 버튼은 다음 조건에서 활성화한다.

- 현재 이미지가 정상 로드됨
- 삭제되지 않은 모든 박스가 유효한 좌표를 가짐
- 삭제되지 않은 모든 박스에 라벨이 지정됨
- 미라벨 회색 후보 박스가 없음

박스가 하나도 없는 이미지도 사용자가 의도적으로 `객체 없음`으로 확정할 수 있어야 한다.

### 빠른 작업 지원

반복 라벨링 속도를 위해 단축키를 제공한다.

- 다음 이미지
- 이전 이미지
- 확정 후 다음 이미지
- 새 박스 그리기 모드
- 선택 박스 삭제
- 라벨 빠른 지정
- Undo/Redo
- 확대/축소/화면 맞춤

단축키는 UI에 과하게 설명하지 말고 메뉴, 툴팁, 도움말 패널에서 확인 가능하게 한다.

## 탐지 후보 생성

탐지 로직은 UI와 분리된 교체 가능한 서비스로 만든다.

권장 인터페이스 책임:

- 이미지 파일 입력
- 원본 이미지 크기 파악
- 후보 bbox 목록 반환
- confidence 반환 가능
- 오류 반환 가능
- 취소 가능

탐지 구현체는 다음 중 하나일 수 있다.

- 로컬 객체 검출 모델
- 이미지 처리 알고리즘 기반 후보 생성
- 개발용 더미 detector
- 수동 라벨링 전용 detector 없음 모드

탐지 모델이 아직 준비되지 않은 상태에서 AI 기능이 있는 것처럼 보이게 만들지 않는다. 모델이 없으면 명확히 더미/수동 모드로 구현하고, 나중에 실제 detector를 붙일 수 있는 구조를 남긴다.

후보 박스는 자동으로 확정하지 않는다. 모든 후보는 사용자가 확인하기 전까지 `proposal` 상태다.

## 데이터 저장

프로젝트 파일은 앱의 작업 상태를 온전히 복원할 수 있어야 한다.

저장해야 할 정보:

- 프로젝트 이름
- 이미지 폴더 경로
- 이미지 목록과 각 이미지의 상대 경로
- 원본 이미지 크기
- 이미지 상태
- 박스 목록
- 박스 좌표
- 박스 상태
- 라벨 목록
- 라벨 색상
- 선택적 detector 메타데이터
- 마지막 저장 버전

원본 이미지는 수정하지 않는다. 프로젝트 데이터는 별도 파일 또는 프로젝트 폴더 안에 저장한다.

좌표 저장 규칙:

- bbox는 원본 이미지 픽셀 기준으로 저장한다.
- x, y는 좌상단 좌표다.
- width, height는 양수다.
- bbox는 이미지 경계를 벗어나면 안 된다.
- 화면 표시용 좌표를 저장하지 않는다.

자동 저장:

- 라벨 변경, 박스 편집, 확정 상태 변경 후 자동 저장한다.
- 저장 실패 시 사용자에게 명확히 알리고 재시도할 수 있게 한다.
- export는 프로젝트 저장과 별개로 동작하지만, export 직전 현재 변경 사항을 저장하거나 저장 실패를 경고한다.

## COCO Export

COCO export는 표준 JSON 구조를 따른다.

필수 최상위 키:

- `images`
- `annotations`
- `categories`

권장 추가 키:

- `info`
- `licenses`

`images` 항목:

- `id`
- `file_name`
- `width`
- `height`

`annotations` 항목:

- `id`
- `image_id`
- `category_id`
- `bbox`: `[x, y, width, height]`
- `area`: `width * height`
- `iscrowd`: `0`

`categories` 항목:

- `id`
- `name`
- `supercategory`

COCO 표준에 없는 앱 전용 정보는 export JSON에 섞지 않는다. 필요한 경우 별도의 sidecar metadata 파일로 저장한다.

Export 규칙:

- 라벨이 지정된 유효 박스만 annotation으로 내보낸다.
- 미라벨 후보 박스는 제외한다.
- 확정되지 않은 이미지도 export 대상에 포함할 수 있다.
- export 전에 미확정 이미지 수, 미라벨 박스 수, 오류 이미지 수를 요약해서 보여준다.
- 사용자가 경고를 확인하면 미완성 상태에서도 export를 계속할 수 있다.
- 파일 경로는 선택한 export 기준에 맞춰 상대 경로를 사용한다.

Export 옵션:

- 전체 이미지 export
- 확정 이미지만 export
- 라벨된 annotation만 export
- 빈 이미지 포함 여부
- 이미지 파일 복사 여부

## 에러 처리

사용자가 막히지 않게 복구 가능한 오류를 설계한다.

처리해야 할 오류:

- 이미지 폴더 접근 불가
- 지원하지 않는 이미지 형식
- 손상된 이미지
- 이미지 디코딩 실패
- detector 실패
- 저장 실패
- export 실패
- 기존 프로젝트 파일 버전 불일치

오류 메시지는 원인, 영향, 다음 행동을 짧게 알려준다.

예:

- "이미지를 열 수 없습니다. 파일이 손상되었거나 지원하지 않는 형식입니다."
- "현재 변경 사항을 저장하지 못했습니다. 권한 또는 디스크 공간을 확인한 뒤 다시 시도하세요."

## 성능 기준

대량 이미지 폴더를 다룰 수 있어야 한다.

- 이미지 목록은 점진적으로 로드한다.
- 썸네일은 캐시한다.
- 원본 이미지는 필요할 때만 디코딩한다.
- 탐지는 큐 기반으로 실행하고 취소 가능해야 한다.
- UI 스레드를 막지 않는다.
- 큰 이미지에서도 박스 편집이 부드럽게 동작해야 한다.

동시에 모든 원본 이미지를 메모리에 올리지 않는다.

## 접근성

- 키보드만으로 기본 라벨링 흐름을 수행할 수 있어야 한다.
- 선택 상태와 오류 상태는 색상만으로 전달하지 않는다.
- 버튼과 아이콘에는 툴팁 또는 접근성 라벨을 제공한다.
- 텍스트는 작은 패널 안에서도 잘리지 않게 한다.
- 고대비 환경에서 박스와 선택 강조가 보여야 한다.

## Flutter 구현 원칙

### 아키텍처

UI, 상태 관리, 파일 I/O, detector, COCO export를 분리한다.

권장 모듈:

- `project`: 프로젝트 생성, 열기, 저장
- `image_import`: 폴더 스캔, 이미지 메타데이터 추출
- `detector`: 후보 박스 생성 인터페이스와 구현체
- `annotation`: 박스, 라벨, 이미지 상태 도메인 모델
- `viewer`: 이미지 표시, 좌표 변환, 오버레이, 편집 인터랙션
- `export`: COCO JSON 생성과 검증
- `settings`: 사용자 설정과 단축키

도메인 모델은 UI 위젯에 종속되지 않게 만든다.

### 상태 관리

상태 관리 방식은 프로젝트 생성 시 하나를 선택하고 일관되게 사용한다. Riverpod, Bloc, Provider 등 무엇을 쓰든 다음 원칙을 지킨다.

- 이미지별 상태와 프로젝트 전체 상태를 분리한다.
- 박스 편집 상태와 저장된 annotation 상태를 구분한다.
- 긴 작업은 취소 가능한 비동기 작업으로 관리한다.
- UI는 도메인 모델을 직접 임의 수정하지 않고 명시적 액션을 통해 변경한다.

### 좌표 변환

이미지 뷰어 구현에서 가장 중요한 부분은 좌표 변환이다.

반드시 분리해서 테스트한다.

- 원본 이미지 좌표 -> 화면 좌표
- 화면 좌표 -> 원본 이미지 좌표
- zoom/pan/fit 상태 반영
- 이미지 경계 clamp
- 최소 박스 크기 보정

좌표 변환 버그는 학습 데이터 품질을 망가뜨리므로 UI 테스트와 단위 테스트를 모두 작성한다.

### 파일과 경로

- 한글 파일명과 공백이 있는 경로를 지원한다.
- 프로젝트 내부에는 가능하면 이미지 폴더 기준 상대 경로를 저장한다.
- 원본 파일 이동 또는 삭제 시 복구/재연결 UX를 제공한다.
- 경로 문자열을 직접 이어 붙이지 말고 플랫폼 경로 API를 사용한다.

## 테스트 기준

### 단위 테스트

- bbox 좌표 clamp
- bbox area 계산
- 라벨 생성/수정/삭제 규칙
- 이미지 확정 가능 조건
- COCO JSON 생성
- 미라벨 박스 export 제외
- 좌표 변환
- 프로젝트 저장/로드 직렬화

### 위젯 테스트

- 이미지 선택 시 박스 목록 갱신
- 박스 목록 선택 시 오버레이 선택 변경
- 라벨 지정 시 박스 색 변경
- 모든 박스 라벨 지정 후 확정 버튼 활성화
- 미라벨 박스가 있으면 확정 버튼 비활성화
- 객체 없음 이미지 확정 가능

### 통합 테스트

- 새 프로젝트 생성 -> 폴더 선택 -> 후보 생성 -> 라벨 지정 -> 확정 -> export
- 미확정 이미지가 남아 있는 상태에서 export
- detector 실패 이미지가 있는 상태에서 export 경고
- 프로젝트 재열기 후 상태 복원

## UI 세부 지침

- 앱 첫 화면은 실제 작업 시작 화면이어야 한다.
- 마케팅용 랜딩 페이지를 만들지 않는다.
- 주요 버튼은 명확한 동사로 표시한다.
- 위험한 작업은 확인을 받는다.
- 삭제는 undo 가능하게 만든다.
- 반복 작업에서 모달을 남발하지 않는다.
- 새 라벨 생성은 라벨 선택 흐름을 끊지 않아야 한다.
- 진행률은 이미지 단위와 전체 단위 모두 보여준다.
- export 전 검증 결과는 요약과 상세를 함께 제공한다.
- 사용자가 지금 어떤 이미지를 편집 중인지 항상 명확해야 한다.

## 데이터 품질 규칙

학습 데이터 품질을 위해 다음을 강제하거나 경고한다.

- width 또는 height가 0 이하인 박스 금지
- 이미지 밖으로 나간 박스 금지
- 라벨 없는 annotation export 금지
- 중복 라벨명 금지
- 너무 작은 박스는 경고
- 동일 이미지 안의 완전 중복 박스는 경고

경고는 사용자의 작업을 막기보다 검토할 수 있게 해야 한다. 단, COCO 형식 자체가 깨지는 오류는 export를 막는다.

## 개발자가 지켜야 할 원칙

- 사용자 반복 작업 속도를 최우선으로 본다.
- AI 후보는 보조 기능이며 사용자의 수정 흐름을 방해하면 안 된다.
- 원본 이미지를 변경하지 않는다.
- COCO export의 좌표와 category id 안정성을 매우 중요하게 다룬다.
- 구현 전에 기존 구조와 테스트를 확인한다.
- 기능 추가 시 저장/로드/export에 미치는 영향을 함께 점검한다.
- 화면 표시 좌표와 원본 이미지 좌표를 혼동하지 않는다.
- 임시 detector를 실제 AI처럼 표현하지 않는다.
- 대용량 이미지 폴더에서도 UI가 멈추지 않게 한다.
- 새 기능에는 사용자가 막히는 빈 상태, 오류 상태, 진행 상태를 함께 설계한다.

## 완료 기준

기능을 완료했다고 말하려면 다음을 확인한다.

- 사용자가 새 프로젝트를 만들 수 있다.
- 이미지 폴더를 선택할 수 있다.
- 이미지 목록과 상태가 표시된다.
- 후보 박스 또는 수동 박스를 만들 수 있다.
- 박스를 선택, 이동, 크기 조절, 삭제할 수 있다.
- 라벨을 생성하고 박스에 지정할 수 있다.
- 라벨 지정 후 박스 색이 변경된다.
- 이미지 확정 조건이 정확히 동작한다.
- 미확정 이미지가 있어도 COCO export가 가능하다.
- export JSON이 COCO 구조와 좌표 규칙을 만족한다.
- 프로젝트를 닫았다가 다시 열어도 상태가 복원된다.
- 핵심 단위 테스트와 주요 사용자 흐름 테스트가 통과한다.

## 용어

- 후보 박스: AI 또는 알고리즘이 객체가 있을 가능성이 있다고 제안한 bbox
- 미라벨 박스: 아직 category가 지정되지 않은 bbox
- 라벨 박스: category가 지정된 bbox
- 확정 이미지: 사용자가 해당 이미지의 박스와 라벨을 검토 완료했다고 표시한 이미지
- COCO export: COCO object detection annotation JSON 생성
